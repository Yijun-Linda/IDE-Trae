## 项目背景

本项目是 **TRAE SOLO 挑战赛「Hello AI 科技致善」公益赛道** 参赛作品，聚焦**心智障碍群体就业支持**命题。

### 核心问题

心智障碍青年（包括唐氏综合征、自闭症等）在就业场景中面临巨大挑战。据实地观察，能在慈善商店正常就业的唐氏综合征或自闭症青年，可能不到该群体的 1%。

### 研究方向演进

**初始方向**：探索如何利用 AI 技术（计算机视觉、语音交互、大语言模型）为心智障碍青年提供实时任务支持

**当前方向**：基于实地采访的深刻洞察，产品定位从"技术增强"转向"隐形辅助"——不是提供一种更好的辅助工具，而是让辅助工具的存在逐渐变得不必要

### 核心洞察

**社会化的本质是不要搞特殊**（来自杨老师，慈善咖啡店店长）

真正的包容不是创造一个特殊的环境，而是把心智障碍者放在真实的世界里，相信他们能够适应。这改变了产品的根本设计哲学：辅助工具必须足够隐形、足够通用，成为工作环境的自然组成部分，而不是让员工显得特殊的标记。

## 产品定义

### 核心功能

AI认知翻译官部署在收银系统和云打印机之间，将外卖/线下订单翻译为心智障碍员工可执行的动作脚本小票。

**为什么是云打印机**：

- 店里本来就有票据打印机，本来就在出小票
- AI只是让小票的内容变得更友好，不增加可见的新设备
- 符合杨老师"不要搞特殊"的原则
- 纸质小票具有物理实体感，员工可以拿着、指着、打勾

### 设计原则

1. **不是加法而是翻译**：改造已有工具（小票）而非插入新设备
2. **结果导向→过程导向**：普通小票显示"1x 拿铁"，AI翻译后显示"拿冰杯→加冰块→加燕麦奶→倒浓缩"
3. **时间延迟自动出单**：比赛MVP使用5分钟时间延迟，v1.0加入绿色按钮让员工控制节奏
4. **简短、流程化、动作指令**：每步不超过10个字，使用动作动词

### 成功标准

不是心智障碍员工使用了多少次辅助工具，而是他们使用辅助工具的频率是否在逐渐降低。**辅助的可消退性比辅助本身更重要**。

## 技术栈

| 组件       | 选型               | 原因             |
| -------- | ---------------- | -------------- |
| **后端**   | Python + Flask     | Webhook接收、API调用 |
| **AI模型** | 通义千问/豆包API    | 订单拆解Prompt工程 |
| **打印机** | 易联云/飞鹅云打印机API | 已对接外卖平台 |
| **硬件**   | 无（v0.1）/ ESP8266无线按钮（v1.0） | 比赛MVP零新增设备，v1.0加入按钮 |
| **开发工具** | Trae SOLO        | 比赛要求           |

**技术细节**：

**识别服务**：当前方案不需要识别服务（计算机视觉、语音识别），订单数据从外卖平台直接以结构化JSON格式传入，AI任务是将结构化数据"翻译"为动作脚本，而非识别图像或语音。

**大模型 vs 规则引擎**：
- v0.1 比赛MVP：使用大模型（通义千问/豆包）快速验证核心假设
- v1.0 生产版本：根据实际使用数据引入规则引擎，降低成本、提高响应速度

**隐私保护**：
- 数据最小化：只处理饮品名称、规格，丢弃顾客联系方式
- 不存储原始数据：订单翻译后立即打印，不持久化存储
- 密钥管理：环境变量存储，`.env`文件不提交到代码仓库

**技术演进说明**：

技术方案从"阿里云视觉智能 + 实时多模态识别"演进为"云打印机 + AI翻译 + 时间延迟"。原因：

1. 采访发现心智障碍员工对"触手可见的提醒"响应最好，耳机等复杂技术反而增加认知负担
2. 云打印机方案不增加可见设备，符合"不要搞特殊"原则
3. 比赛MVP使用5分钟时间延迟控制出单节奏，v1.0加入绿色按钮让员工控制节奏
4. 产品成功不依赖技术复杂度，而依赖对认知约束的深刻理解

## 核心洞察摘要

### 来自杨老师的五个真实需求

1. **流程拆解与步骤化引导**：咖啡师岗位需要同时处理多维度信息（冷热、冰量、杯型、堂食/外带、制作顺序）
2. **多任务排序与优先级管理**：多个客人同时点单时容易花，需要明确的顺序指引
3. **环境状态识别与检查清单**：关机问题反复出现，需要可感知的检查点而非抽象概念
4. **沟通辅助与需求表达**：引导客人做选择的话术模板
5. **辅导员远程支持与异步协作**：减少同步打断式支持

### 设计约束

- 教他们一定要简短、流程化、短语式，最好是动作指令
- 只要不是触手可见的提醒，他们都很难主动去查阅
- 职业康复不等于就业，零支持的百分百脱手几乎不可能
- 优化的目的是没有优化
