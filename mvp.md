# AI认知翻译官：心智障碍咖啡店订单处理系统

## 项目信息

- **项目名称**：AI认知翻译官（AI Cognitive Translator）
- **目标用户**：上海塘桥街道慈善咖啡店心智障碍员工及辅导员杨老师
- **核心场景**：将外卖/线下订单翻译为心智障碍员工可执行的动作脚本
- **生成日期**：2026-04-23

***

## 一、产品定义

### 1.1 核心问题

心智障碍员工在咖啡店工作时面临的核心障碍：

- **结果→过程的认知转换困难**：普通小票显示"1x 拿铁（去冰，燕麦奶）"，员工需要自己拆解成动作步骤
- **多订单并行导致认知超载**：当多个订单同时涌入，工作记忆通道被堵塞，容易搞混、重复做
- **异常处理缺乏物理锚定**：原料缺货、设备故障时，员工不知道如何应对

### 1.2 解决方案

AI部署在收银系统和云打印机之间，作为认知翻译层：

- **输入**：外卖平台/线下收银系统的原始订单数据
- **处理**：AI将结果导向的订单翻译为过程导向的动作脚本
- **输出**：通过云打印机API输出重构后的小票

**关键创新**：不是给员工增加新设备，而是改造他们已经在使用的工具（小票），把世界翻译成他们能懂的语言。

### 1.3 产品形态

**硬件层**：
- 云打印机（已存在，对接美团/饿了么等平台）
- 一个无线按钮（绿色=完成）

**软件层**：
- AI认知翻译引擎（Webhook接收订单 → Prompt工程拆解步骤 → 调用打印机API）
- 打印队列管理（控制出票节奏）

**交互层**：
- 员工：看小票、做咖啡、按绿色按钮
- 杨老师：处理异常、配置规则

## 二、交付物

### 2.1 核心交付物

**1. AI认知翻译引擎**
- 接收外卖平台Webhook
- 调用大模型API（如通义千问/豆包）进行订单拆解
- 调用云打印机API输出动作脚本小票

**2. 动作脚本小票模板**
```
========================================
订单 #{order_id} | 第 {current}/{total} 杯
========================================

【动作脚本】
□ {步骤1}
□ {步骤2}
□ {步骤3}

========================================
按绿色按钮 → 完成，出下一杯
========================================
```

**3. 物理按钮系统**
- 绿色按钮：员工完成一杯后按一下，触发下一单打印

**4. 对比展示材料**
- 普通小票 vs AI翻译后小票的对比图
- 制作流程对比视频（使用前 vs 使用后）

### 2.2 辅助交付物

**1. 异常处理机制**
- 自动切换回原始小票模式（当AI服务异常时）

**2. 杨老师配置界面**
- 步骤模板配置（每个饮品的标准步骤）
- 员工能力配置（哪些员工需要更多步骤拆解）

## 三、成功标准

### 3.1 用户层面

**心智障碍员工**：
- [ ] 能独立完成单杯饮品的制作（无需杨老师口头指导）
- [ ] 多订单场景下的错误率降低50%以上
- [ ] 制作时间接近普通员工（差距在20%以内）

**杨老师（辅导员）**：
- [ ] 同步打断式支持减少50%以上
- [ ] 能灵活配置不同员工的步骤模板
- [ ] 异常处理时间缩短（从发现问题到解决问题）

### 3.2 系统层面

- [ ] 订单翻译准确率 > 95%（步骤拆解正确）
- [ ] 打印延迟 < 3秒（从接单到出票）
- [ ] 系统可用性 > 99%（不依赖复杂硬件）

### 3.3 社会价值层面

- [ ] 不增加员工的社会特殊感（外观与普通咖啡店无区别）
- [ ] 实施成本可控（慈善咖啡店可承受）
- [ ] 可复制到其他类似场景（其他慈善咖啡店、烘焙坊等）

## 四、设计原则（引用全局认知体系）

本项目应用以下认知约束：

1. **工作记忆是稀缺资源**（rules/principles/INDEX.md）
   - 一次只处理一杯，避免认知超载
   - 动作脚本不超过3步

2. **被动接收优于主动检索**（field_visit_tang_coffee_summary.md）
   - 小票自动打印，无需员工主动获取信息
   - 物理按钮简化交互

3. **不要搞特殊**（field_visit_tang_coffee_summary.md）
   - 不增加可见的新设备
   - 改造已有工具而非插入新工具

4. **结构同构性思维**（rules/principles/INDEX.md）
   - 将复杂订单结构同构为简单动作序列
   - 保留原始信息的同时降低认知维度

## 五、技术栈

- **后端**：Python/Node.js（Webhook接收、API调用）
- **AI模型**：通义千问/豆包（订单拆解Prompt工程）

- **硬件**：无线按钮（ESP8266/蓝牙模块）
- **部署**：阿里云/腾讯云函数（低成本、免运维）
