## Step 1: Make Requirements Less Dumb（质疑需求）

### 1.1 原始需求回顾

从 mvp.md 中提取的核心需求：

1. **核心问题**：结果→过程的认知转换困难、多订单并行导致认知超载、异常处理缺乏物理锚定
2. **解决方案**：AI部署在收银系统和云打印机之间，将订单翻译为动作脚本小票
3. **关键创新**：不是给员工增加新设备，而是改造他们已经在使用的工具（小票）
4. **成功标准**：员工能独立完成单杯制作、错误率降低50%、制作时间接近普通员工

### 1.2 需求质疑

**质疑一：异常处理缺乏物理锚定，这个需求是否真实存在？**

根据 field_visit_tang_coffee_summary.md，杨老师处理异常的方式是：电话打过来了，老师，这个怎么处理？她必须放下饭盒，噔噔噔跑回来教。

异常处理的核心不是缺乏物理锚定，而是**员工在不确定时知道该找谁、能找到谁**。小票底部预留异常处理区可能并不能解决这个问题，因为员工可能想不起来去看小票底部。

**质疑二：成功标准中的"错误率降低50%"是否可测量？**

在18天的比赛周期内，很难收集到足够的基线数据（使用前错误率）和对比数据（使用后错误率）。这个标准可能无法验证。

**质疑三："改造已有工具而非插入新设备"这个原则是否被严格执行？**

当前方案仍然增加了一个无线按钮。虽然按钮比耳机简单，但它仍然是一个新设备。员工需要学习"按绿色按钮出下一单"这个动作。

### 1.3 需求修正建议

1. **删除"异常处理缺乏物理锚定"作为核心问题**：异常处理回归到杨老师的人工支持，系统不负责解决异常，只负责在AI服务异常时自动回退到原始小票模式。

2. **将"错误率降低50%"改为可验证的标准**：比如"杨老师同步打断式支持减少50%"，这个可以通过杨老师的主观反馈来验证。

3. **重新审视按钮的必要性**：是否可以用更简单的机制替代按钮？比如时间延迟（假设每杯制作需要固定时间，时间到自动出下一单）？

## Step 3: Simplify or Optimize（简化优化）

### 3.1 简化机会识别

**机会一：简化按钮交互**

当前设计：员工完成一杯后按绿色按钮，打印机出下一单。

简化方案：删除按钮，使用**时间延迟自动出单**。

- 假设每杯制作需要3-5分钟
- 系统打印第一单后，等待5分钟自动出第二单
- 如果员工做得快，可以手动按打印机上的"进纸"按钮（已有功能）来提前出单

优点：
- 不需要额外的无线按钮硬件
- 不需要对接按钮API
- 员工不需要学习新动作

缺点：
- 如果员工做得慢，5分钟到了还没完成，新单会提前出来
- 需要测试时间延迟的合理性

**机会二：简化AI翻译引擎**

当前设计：调用大模型API（通义千问/豆包）进行订单拆解。

简化方案：使用**规则引擎替代大模型**。

- 预设10-15种常见饮品的步骤模板
- 用简单的字符串匹配和替换完成翻译
- 例如：如果订单包含"拿铁"和"去冰"，就套用"拿铁-去冰"模板

优点：
- 不需要调用大模型API，降低成本和延迟
- 翻译结果完全可控，不会出现意外输出
- 实现简单，几小时即可完成

缺点：
- 无法处理未预设的饮品类型
- 灵活性不如大模型

**机会三：简化小票模板**

当前模板：
```
========================================
订单 #{order_id} | 第 {current}/{total} 杯
========================================

【动作脚本】
□ {步骤1}
□ {步骤2}
□ {步骤3}

========================================
按绿色按钮 → 完成，出下一杯
========================================
```

简化模板：
```
第 {current}/{total} 杯
□ {步骤1}
□ {步骤2}
□ {步骤3}
```

- 删除订单号（员工不需要知道）
- 删除按钮说明（如果删除按钮）
- 删除装饰线（节省纸张）

### 3.2 推荐的最简方案

**方案A：最简可行方案（推荐用于v0.1验证）**

- 规则引擎替代大模型
- 时间延迟自动出单（无需按钮）
- 极简小票模板

**方案B：平衡方案（推荐用于v1.0生产）**

- 大模型API进行订单拆解
- 无线按钮控制节奏
- 完整小票模板

## Step 5: Automate（自动化）

### 5.1 自动化时机判断

根据 Elon Algorithm，自动化是最后一步，过早自动化会导致资本浪费。

当前阶段（比赛开发阶段）**不应该追求自动化**：
- 系统尚未稳定，自动化会加速错误
- 18天周期内，手动操作更高效
- 比赛评审关注的是产品价值，不是自动化程度

### 5.2 可以自动化的环节（v1.0之后）

**自动化点一：订单翻译**

当Prompt模板稳定后，可以自动化整个翻译流程：
- Webhook接收订单 → AI翻译 → 调用打印机API
- 无需人工干预

**自动化点二：异常检测**

- 监控AI服务健康状态
- 异常时自动切换回原始小票模式
- 发送通知给杨老师

**自动化点三：数据收集**

- 自动记录每杯制作时间
- 自动统计错误率
- 生成周报给杨老师

### 5.3 比赛阶段的手动操作

- **订单翻译**：手动触发（通过测试脚本）
- **打印队列**：手动管理（通过按钮或时间延迟）
- **异常处理**：手动切换（如果AI服务异常，手动重启服务）
- **数据收集**：手动记录（观察并记录员工使用情况）

## 最简可行方案（MVP）

如果时间和资源极度受限，以下是可以在3天内完成的最简方案：

**硬件**：云打印机（已存在）

**软件**：
- 规则引擎（10种饮品的硬编码模板）
- 时间延迟自动出单（5分钟延迟）
- 无异常回退（AI服务异常时手动重启）

**小票模板**：
```
第 X/Y 杯
□ 步骤1
□ 步骤2
□ 步骤3
```

**交付物**：
- 可以工作的AI翻译+打印系统
- 普通小票 vs AI小票的对比图
- 使用说明（给杨老师）

这个方案虽然简单，但已经能验证核心假设：**把小票从结果导向翻译为过程导向，是否能帮助心智障碍员工独立完成制作？**
