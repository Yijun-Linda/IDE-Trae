## 任务描述

设计Prompt模板，将咖啡订单翻译为心智障碍员工可执行的动作脚本。支持10个SKU（5热饮+5冷饮）。

### SKU列表

**热饮（5个）**：
1. 美式咖啡
2. 拿铁
3. 卡布奇诺
4. 摩卡
5. 焦糖玛奇朵

**冷饮（5个）**：
1. 冰美式
2. 冰拿铁
3. 冰卡布奇诺
4. 冰摩卡
5. 冰焦糖玛奇朵

### 规格支持
- 杯型：中杯/大杯
- 牛奶：全脂/燕麦/脱脂
- 甜度：无糖/半糖/全糖
- 冰量：正常冰/少冰/去冰
- 加料：加浓缩/加糖浆

## 认知约束

### 第一性原理
- 问：为什么要用AI而不是硬编码？
- 答：因为订单组合太多（10SKU × 多规格），硬编码维护成本高，AI可以灵活处理

### 结构同构性思维
- 将自然语言订单结构同构为步骤化动作序列
- 示例："1x 拿铁（去冰，燕麦奶）" → "拿冰杯→加冰块→加燕麦奶→倒浓缩"

### 70%问题解决
- 为AI打开感知通道：提供Few-shot示例
- 建立清晰成功标准：定义准确率>80%的验收条件

## 技术细节

### Prompt设计原则

**1. 角色定义**
```
你是一个咖啡制作指导助手。你的任务是将顾客的订单翻译为心智障碍员工可以一步一步执行的咖啡制作步骤。
```

**2. 输出格式约束**
```
输出必须是JSON格式：
{
  "steps": ["步骤1", "步骤2", "步骤3"],
  "total_steps": 3,
  "drink_name": "饮品名称",
  "notes": "注意事项（可选）"
}
```

**3. 步骤设计原则**
- 每步不超过10个字
- 使用动作指令：拿、加、倒、盖、按
- 避免抽象词：适量、少许、一些
- 按时间顺序排列

### Few-shot示例

**示例1：热拿铁**
```
输入：1x 拿铁（大杯，燕麦奶）
输出：
{
  "steps": ["拿热杯", "加燕麦奶", "萃取浓缩", "倒入混合", "盖盖出杯"],
  "total_steps": 5,
  "drink_name": "热拿铁",
  "notes": "大杯"
}
```

**示例2：冰美式**
```
输入：1x 冰美式（中杯，少冰）
输出：
{
  "steps": ["拿冰杯", "加少量冰", "加水八分满", "萃取浓缩倒入", "盖盖出杯"],
  "total_steps": 5,
  "drink_name": "冰美式",
  "notes": "少冰"
}
```

### 风险与应对
| 风险 | 应对措施 |
|------|---------|
| AI响应不稳定 | 添加重试机制，3次失败回退到原始模式 |
| 响应延迟过高 | 使用轻量级模型（qwen-turbo），目标<2秒 |
| 某些规格理解错误 | 收集错误案例，迭代优化Prompt |
